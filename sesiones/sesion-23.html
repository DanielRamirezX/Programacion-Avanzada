La puerta con
candado.

Tu base de datos es la bóveda del proyecto. Hoy aprendes a abrirla con llave — placeholders y sentencias parametrizadas — y a reconocer al que llega a forzar la cerradura con un simple ' OR 1=1 --.

I · El dato que viene de fuera

Toda entrada de usuario es
culpable hasta demostrar lo contrario.

Ya conectas a la base de datos y sabes hacer SELECT e INSERT. Pero hay una diferencia abismal entre una consulta que funciona en tu compu con datos de prueba y una que sobrevive a un usuario malicioso. Esa diferencia se llama SQL injection, y es el ataque número uno contra aplicaciones desde hace más de dos décadas. La regla de hoy es una sola, y no admite excepciones: nunca pegues lo que el usuario escribió directamente dentro del texto de tu SQL.

Lo bueno: defenderse es más fácil que atacar. Una sola técnica — las sentencias parametrizadas — cierra la puerta por completo. La vemos hoy en Python (placeholders ? / %s) y en Java (PreparedStatement), y la dejamos instalada en la capa de datos de tu proyecto.

IV · La defensa — Python ↔ Java

La llave es la misma idea
en los dos lenguajes.

La sentencia parametrizada separa la consulta (la frase fija, con huecos) de los datos (lo que va en los huecos). El motor de la BD nunca confunde uno con otro — por eso un dato jamás se ejecuta como código. Recorre las operaciones de tu proyecto en Python (base del curso) y su gemelo en Java:

Java JDBC · PreparedStatement

V · ORM vs SQL directo

Dos caminos a la BD.
Los dos pueden ser seguros — si los usas bien.

Hasta ahora escribiste SQL a mano. Existe otra vía: un ORM (Object-Relational Mapping) que traduce objetos de Python a filas de la tabla, sin que escribas SQL. En Python el más usado es SQLAlchemy. La buena noticia de seguridad: un ORM parametriza por ti — siempre. Compara la misma operación en los dos estilos:

SQL directo · sqlite3
ORM · SQLAlchemy

¿Cuál elegir para tu proyecto? Para esta materia, escribir SQL directo y parametrizado te hace entender lo que pasa por debajo — y eso es lo que evaluaré. El ORM es la herramienta profesional para proyectos grandes; conviene que sepas que existe y por qué su parametrización automática es una de sus mayores ventajas de seguridad. Si la IA te genera un ORM, tú debes poder explicar qué SQL produce.

VI · La regla de oro — NO / SÍ

Si tu dato toca el SQL con un + o un f-string,
ya perdiste.

Esta es la frontera entre código que se rompe y código que aguanta. Memorízala: el dato del usuario nunca entra al texto de la consulta — entra aparte, como argumento.

✗ Nunca · vulnerable
sql = f"SELECT * FROM huesped WHERE id = {id}" cursor.execute(sql) # el dato se vuelve parte de la frase # → un ' lo rompe y reescribe todo
Concatenar o interpolar (f-string, +, %, .format()) mezcla datos con código. Una comilla del atacante cambia el significado de la consulta. Esto reprueba la entrega.
✓ Siempre · seguro
sql = "SELECT * FROM huesped WHERE id = ?" cursor.execute(sql, (id,)) # la frase es fija; el dato va aparte # → la BD lo trata como texto, jamás código
El placeholder (? en sqlite, %s en psycopg2/MySQL) deja un hueco. El valor viaja en la tupla de parámetros, separado. La BD lo escapa por ti, siempre, sin que pienses en ello.
Nota de seguridad · del Maestro en Seguridad de la Información

Detalle que cuesta vidas en producción: la tupla de un solo elemento lleva coma — (id,), no (id). Sin la coma, Python no ve una tupla y la llamada falla. Y nunca, jamás, uses el operador % de formateo de cadenas para "meter" el dato: %s dentro de las comillas del SQL es un placeholder del driver, no un formateo de Python. Son cosas distintas que se parecen — y confundirlas abre el agujero.

VII · Puente a la Práctica institucional 6

El hotel no es un ejercicio.
Es una bóveda con datos de personas reales.

Tu Práctica 6 — el sistema del hotel — toca la base de datos en cada operación: registrar un huésped, buscar una reservación, listar habitaciones. Cada una de esas consultas recibe datos de fuera. El requisito no es opcional: todo SELECT e INSERT contra la BD del hotel debe usar PreparedStatement (o el placeholder equivalente), para evitar inyección SQL y proteger la integridad de los datos.

Requerimiento A
Registrar huésped
INSERT del huésped con todos sus datos vía PreparedStatement. Nombre con apóstrofo (O'Brien) incluido — sin romper nada.
Requerimiento B
Buscar reservación
SELECT por folio o por nombre, parametrizado. El folio que teclea el recepcionista jamás entra concatenado.
Requerimiento C
Integridad de datos
Ninguna entrada del usuario puede alterar, borrar o exponer filas que no le corresponden. La parametrización lo garantiza.
El caso · cómo se ve en Java

En la capa de datos del hotel, una búsqueda de reservación se ve así: preparas la frase con un ?, asignas el folio con setInt(1, folio), y ejecutas. El recepcionista puede teclear lo que sea en el cuadro de búsqueda — el motor solo buscará ese folio literal, nunca ejecutará lo que venga disfrazado de comando.

La defensa del proyecto será exactamente esta demostración: tú, en clase, mostrando que tu SELECT/INSERT resisten una entrada maliciosa. Si la IA te ayudó a escribir la capa de datos, tú la defiendes línea por línea.
