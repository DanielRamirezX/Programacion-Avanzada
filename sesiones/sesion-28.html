Demo Day.

El día que dejas de explicar lo que vas a construir y demuestras lo que construiste. Un proyecto full-stack vivo: clases, base de datos, una API que responde y un endpoint con IA. Hoy lo defiendes — y se cierra el cuatrimestre.

I · El día llegó

No es una entrega más.
Es el día que defiendes lo que es tuyo.

El Demo Day es la presentación final del proyecto integrador: una demostración en vivo seguida de una defensa oral con mis preguntas. Tu proyecto debe demostrar los cuatro paradigmas del curso trabajando juntos — arquitectura OOP, base de datos integrada, API REST funcional y al menos un endpoint con IA. No vienes a leer diapositivas: vienes a correr tu sistema y a responder por cada línea.

Cuatro paradigmas, un solo proyecto. Hoy se ven trabajando al mismo tiempo: el objeto que guardas en la BD, que expones por la API, que la IA usa para decidir. Eso es lo que defiendes.
OOP BD API REST IA

La defensa final del Demo Day vale el 20% de tu calificación final. No es el código lo único que se evalúa hoy: es tu capacidad de explicarlo, justificarlo y sostenerlo bajo preguntas. Ese es el verdadero entregable de un ingeniero.

IV · Las preguntas que voy a hacer

La defensa empieza
donde termina la demo.

Cuando bajes el telón, vienen las preguntas. No son trampas: buscan confirmar que entiendes lo que construiste. Aquí tienes el banco real — toca cada tarjeta para ver la pregunta completa y cómo prepararla. Si puedes responder estas seis, puedes responder casi cualquier cosa:

Arquitectura · OOP
Por qué esta clase y no una función suelta? Qué encapsula?
Qué busco: que justifiques tu diseño, no que lo recites. Prepárate para señalar una clase concreta y explicar qué responsabilidad única tiene, qué atributos protege y por qué son privados.prepárate: abre tu diagrama de clases y ten lista UNA decisión de diseño que defenderías.
Persistencia · BD
Qué pasa con tus datos cuando apagas el servidor?
Qué busco: que distingas estado en memoria de estado persistido. Muéstrame la tabla, una consulta SELECT real y explica cómo un objeto Python se vuelve fila y vuelve a ser objeto.prepárate: ten un cliente de BD abierto con datos cargados, listo para consultar en vivo.
API REST
Qué código HTTP devuelve este endpoint si le mando datos inválidos?
Qué busco: que conozcas tu contrato. 200/201 en éxito, 400 en datos malos, 404 si no existe, 422 de validación. Demuéstralo desde Swagger mandando una petición que falle a propósito.prepárate: ten en /docs una petición "buena" y una "mala" guardadas.
Inteligencia Artificial
Qué hace tu endpoint de IA exactamente, y qué NO hace?
Qué busco: honestidad técnica. Explica qué modelo o servicio usas, qué entra, qué sale y dónde están sus límites. Una IA con criterio sabe decir "esto lo decide el modelo, esto lo valido yo".prepárate: ten una entrada de ejemplo y di explícitamente qué pasa si el modelo se equivoca.
Integración
Sigue el camino de UN dato: del clic del usuario a la respuesta.
Qué busco: que veas tu sistema como un todo. Petición → ruta de la API → objeto del dominio → consulta a la BD → (si aplica) IA → respuesta JSON. Si puedes narrar ese viaje, dominas la arquitectura.prepárate: ensaya esta narración en voz alta una vez. Es la pregunta estrella.
Uso responsable de IA
Qué parte de este código escribió la IA, y la entiendes?
Qué busco: que defiendas tu código sin importar quién lo tecleó. Te pediré abrir un archivo al azar y explicar una función línea por línea. Si la IA lo hizo, tú lo defiendes.prepárate: si hay código que no entiendes, hoy es tarde para pedírselo a la IA — pídele que te lo explique ANTES.

V · Que no se caiga la demo

Lo que puede fallar, fallará
— a menos que tengas plan B.

El demo en vivo es lo más impresionante y lo más frágil. El wifi se cae, la API se cuelga, la BD está vacía. Los profesionales no improvisan: preparan. Aquí lo que separa una demo sólida de un naufragio:

✗ Lo que hunde una demo
✗Empezar a instalar dependencias o correr migraciones frente a todos.
✗Una base de datos vacía: no hay nada que mostrar en los reportes.
✗Depender del wifi del campus para una API o un modelo en la nube.
✗Buscar a ciegas el endpoint correcto mientras el reloj corre.
✗Improvisar datos de prueba y que la IA reciba basura en vivo.
✓ Plan B de profesional
✓Llega con todo corriendo: servidor levantado, /docs ya abierto.
✓Carga datos de prueba (seed) realistas antes de empezar.
✓Ten un video corto o capturas de respaldo por si algo cae.
✓Prepara peticiones de ejemplo guardadas (buena y mala) en Swagger.
✓Prueba la demo completa una vez anoche en las mismas condiciones.

Regla de oro: si algo falla en vivo, no te paralices. Di "tengo un respaldo de esto", muéstralo y sigue. Saber recuperarte de un error suma más que una demo perfecta sin alma.

Antes de presentar,
todo debe estar en su lugar.

El Demo Day no es solo hablar: hay un entregable. Repo en GitHub, proyecto corriendo y documentación mínima. Marca tu progreso — si todo está en verde, llegas listo a defender:

Listo para defender: 0 de 12 elementos
El código · los 4 pilares
OOP: clases con responsabilidad clara, atributos privados y métodos públicos como puerta de entrada
BD integrada: tus objetos se guardan y se leen de la base; no todo vive en memoria
API REST: endpoints con verbos correctos (GET/POST) y códigos HTTP coherentes
Endpoint de IA: al menos uno funcional, con entrada y salida claras
El repositorio · GitHub
README con descripción, cómo instalar y cómo correr el proyecto
requirements.txt (o equivalente) — que cualquiera pueda levantarlo
.gitignore sin secretos ni claves de API subidas al repo

VIII · Lo que te llevas del curso

Empezaste con un print.
Terminas con un sistema completo.

Antes de las calificaciones, un momento de reflexión. Este proyecto no son cuatro temas sueltos: es la prueba de que sabes integrar. Toca cada pilar y mira cuánto camino recorriste — y cómo se conectan en tu Demo Day:

OOP
de variables sueltas → a objetos con responsabilidad
Base de datos
de listas en memoria → a datos que perduran
API REST
de un script local → a un servicio que el mundo consume
IA con criterio
de copiar respuestas → a usarla como herramienta

La IA seguirá a tu lado toda tu carrera. La diferencia entre quien la usa con criterio y quien depende de ella es exactamente lo que el Demo Day mide: ¿puedes defender lo que entregas? Si la respuesta es sí, este curso cumplió su misión.
