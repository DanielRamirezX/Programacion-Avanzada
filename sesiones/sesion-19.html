Cuando todo
sale mal.

Un archivo que no existe, una división entre cero, una red que se cae. El código siempre puede fallar. Hoy aprendes a atrapar la caída — y a dejar todo limpio aunque el mundo se rompa.

I · El programa optimista vive poco

Asumir que todo saldrá bien
es la forma más rápida de crashear.

Tu código vive en un mundo hostil: el usuario teclea letras donde pedías números, el archivo se borró, el servidor no responde. A esos eventos que rompen el flujo normal los llamamos excepciones. Manejarlas no es opcional: es la diferencia entre un programa que avisa "no encontré el archivo" y uno que se cae con un muro de texto rojo. Hoy vemos las tres palabras que lo controlan — try, except y finally — y cómo cada lenguaje resuelve el mismo problema.

II · El flujo — try, except, finally

Intenta. Si algo truena, atrápalo.
Y pase lo que pase, finally siempre corre.

Las tres palabras forman un guion con dos finales posibles. En el bloque try pones lo que podría fallar; en except, el plan B si falla; en finally, lo que se ejecuta sí o sí — falle o no falle. Pulsa el toggle y observa qué bloques corren en cada caso:

III · El error que sube por el stack

Un error no atrapado
no se queda quieto: sube.

Cuando una función lanza una excepción y no la atrapa, el error brinca a quien la llamó. Y si esa tampoco lo atrapa, sigue subiendo… hasta que alguien lo captura con un try — o hasta que llega arriba del todo y el programa crashea. V · Atrapar bien vs atrapar todo

El except desnudo
es esconder el problema, no resolverlo.

El error más común al empezar: poner un except: sin nombre que se traga cualquier excepción. Parece que "maneja errores", pero en realidad oculta bugs reales — incluso los que escribiste tú. La regla de oro: atrapa solo la excepción que sabes manejar, y deja subir las demás. Compara:

así NO ✗
try: n = int(dato) total = saldo / n except: # desnudo print("hubo un error")
Se traga todo: el ValueError del texto, el ZeroDivisionError y hasta un error de dedo tuyo. Mismo mensaje vago para todo. Imposible saber qué pasó realmente.
así SÍ ✓
try: n = int(dato) total = saldo / n except ValueError: print("escribe un número") except ZeroDivisionError: print("no puede ser cero")
Cada error, su mensaje: sabes exactamente qué falló y le das al usuario una pista útil. Y si aparece un error que NO esperabas, sube — y te enteras de que existe en vez de ocultarlo.

¿Y cuando de verdad quieres un plan B genérico? Atrapa except Exception as e (no el desnudo) y al menos imprime e: así el plan B existe, pero el error no desaparece sin dejar rastro.

Un sistema de reservaciones

La Práctica institucional 5 aterriza todo esto en un caso real. No la entregas hoy, pero hoy aprendes la herramienta exacta que necesita. Lee el encargo:

Practicario MCT158 · Práctica 5 de 6 · Excepciones

La cadena de cines "Cine éxito" quiere un sistema de reservaciones automatizado. El cliente debe poder seleccionar una función, elegir el número de boletos y que el sistema le asigne asientos. El programa se desarrolla con orientación a objetos y debe ser robusto: ninguna entrada del usuario ni ningún imprevisto puede tumbar el sistema.

Por eso, todas las operaciones riesgosas deben envolverse con manejo de excepciones (try/catch/finally): que el usuario teclee una función inexistente, que pida más boletos que asientos disponibles, o que el número de boletos no sea un entero válido. En cada caso el sistema avisa con claridad y sigue funcionando.

Riesgo 01
Función inválida
El usuario elige una función que no existe. Atrápalo y vuelve a pedir — no crashees.
Riesgo 02
Boletos no numéricos
Teclea "tres" en vez de 3. Un ValueError / NumberFormatException que debes manejar.
Riesgo 03
Sin asientos
Pide más boletos que asientos libres. Lanza tu propia excepción y avisa cuántos quedan.

El finally de tu sistema cierra la sesión de reserva: libere o no asientos, siempre deja la taquilla lista para el siguiente cliente.

VIII · La discusión · ¿qué paradigma prefieres?

No hay un ganador absoluto.
Hay trade-offs — y tú eliges con argumentos.

Excepciones (Python/Java/JS) vs errores como valor (Go): cada uno tiene defensores apasionados. La actividad evaluable de hoy es una discusión grupal donde defenderás tu postura. Llega con opinión formada. Estas preguntas te preparan — las respuestas las construyes tú:

Qué es más legible: un bloque try grande, o un if err != nil tras cada llamada?
Go te obliga a ver cada error; Python te deja "olvidarlos". ¿Disciplina o ruido?
Dónde es más fácil ignorar un error por accidente?
En Python un except: desnudo lo entierra. En Go ignorar err es visible (un _ explícito).
Qué pasa con la limpieza de recursos? finally vs defer de Go.
Go no tiene finally: usa defer. Investiga cómo logra el mismo "pase lo que pase".
Para tu proyecto integrador, qué estilo cabe mejor — y por qué?
Python es tu base, así que excepciones. Pero entender Go te hace mejor con las dos.
