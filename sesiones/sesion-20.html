I · Por qué clasificar el error

Un error sin nombre es un fantasma.
Un error clasificado es un plan.

Cuando tu programa explota, no explota "porque sí": lanza un objeto excepción de un tipo concreto. Ese tipo es información. ValueError te dice algo muy distinto que FileNotFoundError, y cada uno pide un manejo distinto. Antes de escribir un solo try/except útil, tienes que saber contra qué peleas. Hoy ordenamos el zoológico de los errores — en Python, tu lenguaje base, con la mirada puesta en Java, donde el compilador toma partido.

El gran corte de Java
Comprobadas
Checked. El compilador no te deja olvidarlas: o las atrapas con try/catch, o declaras throws. Suelen ser fallos del entorno: archivos, red, base de datos.
El gran corte de Java
No comprobadas
Unchecked. El compilador las deja pasar; estallan en ejecución. Suelen ser bugs: índices fuera de rango, null, conversiones imposibles. Python entero vive aquí.

II · La jerarquía built-in de Python

Toda excepción de Python
desciende de un solo ancestro.

En Python las excepciones son clases, y forman un árbol de herencia que arranca en BaseException. Casi todo lo que atraparás cuelga de Exception. Entender este árbol no es trivia: cuando escribes except Exception, atrapas a todos los descendientes — por eso atrapar muy arriba es peligroso. Toca cada nodo para ver de qué hereda:

La lección de diseño: atrapa lo más específico posible. Si solo te interesa una división entre cero, atrapa ZeroDivisionError, no ArithmeticError ni mucho menos Exception. Cuanto más arriba atrapas, más errores ocultas — incluidos los que no esperabas.

IV · ¿Te obliga el compilador? — checked vs unchecked en Java

En Java, el error tiene
un juez: el compilador.

Aquí está la gran diferencia con Python. Java parte sus excepciones en dos: las comprobadas (checked), que el compilador te obliga a manejar, y las no comprobadas (unchecked), que deja correr. La regla técnica: todo lo que hereda de Exception pero no de RuntimeException es checked. Pulsa cada caso y mira el veredicto del compilador:

¿Y en Python? No hay checked: todas las excepciones son, en términos de Java, unchecked. Python confía en ti — nunca te obliga a manejar nada. Más libertad, más responsabilidad: nadie te recordará que ese archivo puede no existir. El recordatorio tienes que ponértelo tú, con el mapa de excepciones que armas hoy.

V · Las reglas de oro del manejo

Atrapar un error es fácil.
Atraparlo bien es el oficio.

Manejar excepciones mal es peor que no manejarlas: un error silenciado es una bomba con temporizador. Estas son las cuatro reglas que separan al código profesional del que "compila pero miente". Toca cada tarjeta — las verdes son lo que SÍ, las rojas lo que NO:

✓Sé específico
Atrapa el tipo exacto que esperas, no Exception a secas. Así, un error que NO previste sigue subiendo y se hace visible — en vez de quedar enterrado.
# sí: atrapas justo lo que puede fallartry: edad = int(dato)except ValueError: edad = 0
✗No silencies el error
El except: pass vacío es el pecado capital. El programa sigue como si nada, pero con datos corruptos. Si atrapas, haz algo: repara, registra o relanza.
# NO: el error desaparece sin rastrotry: procesar(pago)except Exception: pass # ¿qué falló? jamás lo sabrás
✗No las uses como if/else
Las excepciones son para lo excepcional, no para el flujo normal. Si un caso es esperado y frecuente, valídalo con un if. Lanzar excepciones por todo es lento y oculta los errores reales.
# NO: control de flujo disfrazadotry: valor = inventario[clave]except KeyError: valor = 0 # mejor: inventario.get(clave, 0)
✓Conserva el contexto
Cuando relances o registres, no pierdas el error original. En Python usa raise ... from e para encadenar la causa; deja que el mensaje cuente qué dato falló.
try: config = leer(ruta)except FileNotFoundError as e: raise ConfigError(ruta) from e

VI · La anatomía del manejo — try / except / else / finally

Cuatro bloques. Cada uno
responde una pregunta distinta.

Ya sabes qué errores existen y cuándo manejarlos. Ahora el cómo, en Python. El bloque completo tiene cuatro partes y la mayoría de la gente solo usa dos — perdiéndose else y finally, que existen por buenas razones. Toca cada línea para ver qué hace y cómo se llama en Java:

try: archivo = open("ventas.csv") total = int(archivo.readline()) except FileNotFoundError: print("No encontré el archivo") except ValueError as e: print(f"Dato inválido: {e}") else: print(f"Total leído: {total}") finally: archivo.close() # pase lo que pase
El bloque vigilado
try
El código que podría fallar. Mantenlo corto: solo lo arriesgado. Toca las otras líneas del código para recorrer los cuatro bloques.

VII · Redefinir un método sin romper la promesa

Cuando sobrescribes un método,
heredas también su contrato de errores.

Ya viste herencia y polimorfismo. Aquí va la regla fina que casi nadie cuenta: si una clase hija redefine (sobrescribe) un método del padre, no puede inventar nuevos enemigos. En Java, un método sobrescrito no puede lanzar excepciones comprobadas más amplias que las del método original — solo las mismas, subtipos, o ninguna. Es la regla de sustitución de Liskov aplicada a errores: quien usa al padre debe poder usar a la hija sin sorpresas.

// PADRE — declara que puede fallar leyendo class Repositorio { Datos leer() throws IOException { ... } } // HIJA — al sobrescribir, el contrato manda class RepoMemoria extends Repositorio { Datos leer() { ... } // ✓ lanza menos }
Permitido
Igual, más específico, o nada.
La hija puede lanzar las mismas excepciones comprobadas, subtipos de ellas (p. ej. FileNotFoundException en lugar de IOException) o ninguna. Lo que no puede es lanzar una checked nueva y más amplia (como SQLException): rompería al código que confía en el contrato del padre.
¿Y en Python? No hay throws, así que el lenguaje no lo verifica — pero la regla de diseño sigue valiendo. Documenta en el docstring qué excepciones lanza cada método, y al sobrescribir respeta esa promesa. Tu proyecto integrador depende de que una subclase sea sustituible por su padre sin volar todo.

VIII · Puente a la Práctica institucional 5

El caso "Cine éxito"
te va a fallar a propósito.

La quinta práctica institucional de MCT158 es, en el fondo, esta sesión hecha código en Java. Te entrega un sistema de venta de boletos que se rompe con entradas reales y te pide identificarlas, clasificarlas (comprobada vs no comprobada) y manejarlas. Es el ensayo de lo que harás en tu proyecto:

Practicario MCT158 · Práctica 5 de 6 · el encargo

El cine "Cine éxito" tiene un sistema de taquilla en Java que captura datos del usuario por teclado: número de boletos, asiento elegido, nombre del cliente. Funciona… hasta que un usuario teclea "dos" donde se esperaba un número, elige el asiento 99 en una sala de 50, o deja un campo vacío.

Tu encargo: identificar las excepciones comunes que el sistema puede lanzar, distinguir las comprobadas de las no comprobadas, y manejarlas para que el cine nunca vea un crash — solo un mensaje claro y otra oportunidad de capturar bien.

Enemigo 01
NumberFormatException
El usuario teclea "dos" y el sistema hace Integer.parseInt. No comprobada. El equivalente del ValueError de Python.
Enemigo 02
ArrayIndexOutOfBounds…
Pide el asiento 99 en un arreglo de 50. No comprobada. Gemelo de IndexError.
Enemigo 03
NullPointerException
Un campo vacío o un objeto sin inicializar. No comprobada. La que más cuesta a los programadores del mundo.

Fíjate: los tres enemigos del cine son no comprobados — el compilador de Java no te avisó de ninguno. Por eso la práctica importa: te entrena el ojo para ver los errores que ninguna herramienta señala. Exactamente lo que necesitas para el mapa de tu proyecto.
