II · El caso — lee antes de programar

Una taquilla que vende
sin tumbarse.

Practicario MCT158 · Práctica 5 de 6 · Excepciones

La empresa "Cine éxito" desea implementar un sistema de reservaciones automatizado para optimizar la experiencia de sus clientes y reducir errores humanos. Hoy las reservaciones se hacen a mano, y eso ha causado largas filas, conflictos por horarios agotados y problemas con la asignación de asientos.

El sistema debe permitir al usuario seleccionar una función (por película, fecha, hora y sala), elegir el número de boletos y asignar los asientos automáticamente según disponibilidad. Y debe administrar con elegancia todo lo que puede salir mal: si un cliente pide más boletos de los disponibles, el sistema notifica el error y le permite modificar su solicitud; si los datos ingresados (cantidad de boletos o asientos) tienen un formato incorrecto, el sistema identifica el problema y solicita correcciones con claridad.

En resumen: todas las operaciones riesgosas se envuelven con manejo de excepciones (try / catch / finally). Ninguna entrada del usuario ni ningún imprevisto puede tumbar la taquilla.

Riesgo 01 · formato
Boletos no numéricos
El cliente teclea "tres" en vez de 3. Al convertir con Integer.parseInt(...) salta un NumberFormatException. Atrápalo y pide el dato de nuevo.
Riesgo 02 · disponibilidad
Sin asientos
Pide más boletos que asientos libres. Lanza tu propia AsientosAgotadosException, avisa cuántos quedan y permítele modificar la solicitud.
Riesgo 03 · datos nulos
Función inexistente
Elige una función que no está en la cartelera. Un NullPointerException al operar sobre null — o un índice de asiento fuera de rango (ArrayIndexOutOfBoundsException).

Y pase lo que pase — venta exitosa o error — el sistema debe cerrar la sesión de reserva y dejar la taquilla lista para el siguiente cliente. Ese "pase lo que pase" tiene nombre: finally.

III · El concepto — try, catch, finally (6.1)

Intenta. Si algo truena, atrápalo.
Y pase lo que pase, finally siempre corre.

Las tres palabras forman un guion con dos finales posibles. En try pones lo que podría fallar; en catch (Python: except), el plan B si falla; en finally, lo que se ejecuta sí o sí. Cambia de lenguaje y de escenario, y ejecuta paso a paso el momento clave de la práctica: convertir el texto que teclea el cliente en un número de boletos.

La diferencia que importa: Python tiene un else (corre solo si el try no falló); Java no lo usa — pone ese código dentro del propio try. Pero en ambos, finally es idéntico: corre siempre, con error o sin él. Ahí cierras la taquilla, el archivo, la conexión — lo que nunca debe quedar abierto.

V · La familia de las excepciones (6.2 · 6.3 · 6.6)

Todas son Throwable.
Pero no todas se atrapan igual.

En Java toda excepción cuelga de un árbol. Saber dónde cuelga te dice si el compilador te obliga a manejarla (comprobada) o si aparece sola en tiempo de ejecución (no comprobada). Pulsa cada nodo: ahí están las tres excepciones comunes que pide la práctica — y el lugar exacto donde vivirá AsientosAgotadosException.

Pulsa una excepción del árbol (o los botones) para leerla.
Las reglas de oro (6.4 · 6.5): ① atrapa solo la excepción que sabes manejar — nada de catch (Exception e) genérico que se traga todo; cada error, su catch y su mensaje. ② finally libera recursos siempre. ③ al redefinir un método (6.5), una subclase no puede lanzar excepciones comprobadas más amplias que las del método base — solo iguales o más específicas.

4
Maneja las excepciones comunes (actividades 7 y 8)

Cada riesgo del caso es una excepción distinta — y cada una merece su propio catch con un mensaje claro (nunca uno genérico que se trague todo). Identifica las categorías (actividad 7): la IOException sería comprobada; las de abajo son no comprobadas, nacen de la lógica y el formato:

// src/SistemaReservas.java — el manejo, en un solo lugar útil public void reservar(Funcion func, String cantidadTexto) { try { int boletos = Integer.parseInt(cantidadTexto); // ← formato int[] lugares = func.reservar(boletos); // ← null? disponibilidad? System.out.println("Reserva confirmada: " + boletos + " boletos"); } catch (NumberFormatException e) { System.out.println("Escribe la cantidad con números, ej. 3"); } catch (NullPointerException e) { System.out.println("Esa función no existe en la cartelera"); } catch (AsientosAgotadosException e) { // ← la tuya (paso 5) System.out.println("Sin lugar. Disponibles: " + e.getDisponibles()); } finally { System.out.println("Sesión de reserva cerrada"); // SIEMPRE } }
La tercera común (ArrayIndexOutOfBoundsException) aparece si al asignar un asiento accedes a un índice fuera del arreglo. Manéjala validando el rango — o atrápala con su propio catch. Es el clásico "me pasé del tamaño del array" que ya conociste en la Práctica 3.
5
Crea tu excepción propia: AsientosAgotadosException (actividad 9 · 6.6)

Cuando ninguna excepción de Java describe tu problema, creas la tuya. Extiende Exception (así es comprobada: quien llame a reservar() está obligado a manejarla) y guarda el dato útil — cuántos asientos quedan — para dárselo al cliente:

// src/AsientosAgotadosException.java — tu excepción personalizada public class AsientosAgotadosException extends Exception { private int disponibles; public AsientosAgotadosException(String mensaje, int disponibles) { super(mensaje); // el mensaje viaja a la clase base this.disponibles = disponibles; } public int getDisponibles() { return disponibles; } } // ...y dentro de Funcion.reservar(), cuando no alcanza: if (cantidad > disponibles()) { throw new AsientosAgotadosException("No hay lugar", disponibles()); }
throw lanza; throws avisa. En el método escribes throw new AsientosAgotadosException(...) para lanzarla, y en la firma pones throws AsientosAgotadosException para avisar que puede salir de ahí. Como extiende Exception, Java no compila hasta que alguien la atrape con un catch — por eso es "comprobada".

Lo de oro: nota que la taquilla nunca se cierra por un error del cliente. Teclee lo que teclee, pida lo que pida, el sistema atrapa, avisa con claridad y queda listo para el siguiente. Eso es robustez — y es lo que separa un programa "a medias" de uno terminado.

IX · Cómo se evalúa — la rúbrica

Tres frentes.
El mismo estándar.

Frente 01
Diseño del sistema
  • Clases y métodos organizados y coherentes con los requerimientos.
  • Nombres de clases, métodos y variables claros y con propósito.
  • Modularidad: el código dividido en unidades lógicas representadas en clases.
Frente 02
Implementación de requerimientos
  • Selección de funciones por película, fecha, hora y sala.
  • Funcionalidad para elegir el número de boletos.
  • Asignación automática de asientos según disponibilidad.
Frente 03
Manejo de excepciones
  • try detecta errores en las operaciones críticas.
  • catch maneja las comunes (NumberFormatException, NullPointerException).
  • finally asegura la liberación de recursos siempre.
  • Se identifican y manejan comprobadas y no comprobadas.
  • Excepción personalizada para la falta de asientos.

La regla de oro no cambia: diagrama y código deben contar la misma historia. Si tu diagrama declara AsientosAgotadosException pero tu reservar() nunca la lanza, o si atrapas todo con un catch (Exception e) genérico, no estás manejando errores — los estás escondiendo.
