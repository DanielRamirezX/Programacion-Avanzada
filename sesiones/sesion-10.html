Muchos caminos,
una palabra.

Diez if-elif-else apilados son una escalera incómoda. Hoy aprendes a decir lo mismo con una sola estructura — y a reconocer cuándo match deja de ser un switch y se vuelve algo mucho más poderoso.

I · La escalera de elif

Cuando comparas una cosa
contra muchos valores, el if-else se cansa.

Imagina traducir un código de error a su mensaje, o un día de la semana a su nombre. Con if-elif-else repites la misma variable y el mismo operador una y otra vez: if dia == 1 ... elif dia == 2 .... Funciona, pero el ojo se pierde y un olvido se cuela fácil. Casi todos los lenguajes inventaron una estructura para el caso "una variable, muchos valores": el switch. Python llegó tarde y, en vez de copiarlo, trajo algo más ambicioso en la versión 3.10: match/case con coincidencia estructural.

Hoy hacemos tres cosas: ver el mismo problema en cuatro formas, entender cuándo conviene cada una, y cerrar con el Sprint 1 Review — la primera vez que tu proyecto se defiende en público.

II · Un problema, cuatro formas

El mismo enrutador de comandos,
escrito en cuatro idiomas.

Un mini-intérprete recibe un comando de texto ("norte", "sur", "salir") y decide qué hacer. Es el caso clásico de selección múltiple. Pulsa cada pestaña y compara: misma lógica, distinta gramática. Python es tu base; los demás te dan perspectiva.

III · El detalle que rompe todo · break y fallthrough

La diferencia más cara
entre lenguajes vive en una palabra.

En Java y C, un switch que olvida el break sigue ejecutando los casos siguientes — es el famoso fallthrough, fuente eterna de bugs. Python y Go lo arreglaron, cada uno a su manera. Mira el contraste:

✗ Java · olvidaste el break
switch (dia) { case 6: msg = "sábado"; // sin break ↓ case 7: msg = "domingo"; break; }
Con dia = 6 esperabas "sábado"… pero el código cae al caso 7 y termina en "domingo". El compilador no se queja. Un bug silencioso.
✓ Go · sin break, salida automática
switch dia { case 6: msg = "sábado" // rompe solo, automático case 7: msg = "domingo" } // fallthrough es OPT-IN
Go invirtió la regla: cada caso rompe solo. Si de verdad quieres caer al siguiente, lo pides explícito con la palabra fallthrough. Lo peligroso ahora es lo intencional.

Python no tiene este problema porque no hay break ni fallthrough: el primer case que coincide gana, ejecuta su bloque y match termina. Limpio por diseño.

IV · Cómo piensa el match · paso a paso

match no compara todo a la vez.
Prueba caso por caso, de arriba hacia abajo.

El orden importa: gana el primer patrón que coincide. Por eso el comodín case _ siempre va al final — si lo pones arriba, se traga todo. Avanza paso a paso y mira cómo el intérprete recorre los casos con el comando "correr al norte":

match comando = "correr al norte"
case "salir":—
case "norte" | "sur":—
case s if "norte" in s:—
case _:—
·
Pulsa Siguiente o Play para ver al intérprete probar cada caso en orden.

Fíjate en el caso 3: case s if "norte" in s usa una guarda (un if extra) y captura el valor en la variable s. Eso ya no es un switch — es coincidencia de patrones. Lo vemos en la siguiente sección.

V · La joya de Python · structural pattern matching

match no solo compara valores.
Compara la forma de los datos.

Aquí Python deja atrás al switch clásico de Java y Go. Un patrón puede desarmar una lista, un diccionario o un objeto, verificar su estructura y capturar sus partes en variables — todo en la misma línea. Toca cada case para entender qué patrón usa:

match evento: case "ping": responder() case ["mover", x, y]: mover_a(x, y) case ["sumar", *nums]: total(nums) case {"tipo": t, "id": n}: procesar(t, n) case Punto(x=px, y=py): dibujar(px, py) case _: ignorar()
Toca un case
Cinco patrones
Cada case de la izquierda usa un tipo distinto de patrón. Tócalos en orden para ver cómo Python desarma listas, diccionarios y objetos — capturando sus partes sobre la marcha.
Las variables en ámbar son capturas: si el patrón coincide, Python las llena por ti.

Esto es lo que ni Java ni Go tenían cuando Python lo lanzó (Java lo está incorporando apenas, con record patterns). Para un proyecto que procesa comandos, mensajes JSON o eventos, match reemplaza decenas de if isinstance(...) por algo que se lee como la estructura del dato.

VII · Taller en clase · reescribe las calificaciones

Esa cadena de elif del proyecto
tiene su gemelo en match.

Recuerda el clasificador de calificaciones que escribimos con if-elif. Hoy lo reescribes con match usando guardas — porque las calificaciones son rangos, no valores exactos. Compara y luego hazlo con tu propio caso:

Antes · cadena de if-elif
def letra(c): if c >= 90: return "A" elif c >= 80: return "B" elif c >= 70: return "C" else: return "Reprobado"
Funciona perfecto. Para rangos, el if-elif sigue siendo natural — no todo necesita match.
Con match · guardas por rango
def letra(c): match c: case n if n >= 90: return "A" case n if n >= 80: return "B" case n if n >= 70: return "C" case _: return "Reprobado"
Mismo resultado, leído como una tabla. La guarda (if n >= 90) es lo que permite rangos dentro de match. El case _ reemplaza al else.

Justifica tu decisión: eso es lo que evalúo, no la moda.
