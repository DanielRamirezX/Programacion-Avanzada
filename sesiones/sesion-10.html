Sesión X · Gestores de dependencias · Maven · Manejo de excepciones · Persistencia: java.sql y MongoDB

Hablar con lo que
puede fallar.

Un programa serio no vive solo: usa librerías que otros escribieron — y para hablar con una base de datos necesita una de ellas. Hoy aprendes a traer esas librerías sin descargarlas a mano (los gestores de dependencias, con Maven al frente) y a sostener tu código cuando el mundo exterior falla: try / catch, mientras guardas datos que sobreviven al apagón — en tu disco y en la nube.

I · El problema · tu programa necesita código que no escribiste

Nadie construye solo.
Todo proyecto serio usa librerías de otros.

Para hablar con SQLite hace falta un .jar; para MongoDB, tres. En el laboratorio los bajamos a mano y los pusimos en el classpath (el classpath es, sencillamente, la lista de carpetas y .jar donde Java busca las clases que usas; si una librería no está ahí, tu código no compila). Funcionó… pero imagina un proyecto real con decenas de librerías, donde cada una necesita otras, y cada cual en una versión compatible. Cazar todo eso a mano no escala — el problema es tan clásico que tiene nombre propio: el "JAR hell", el infierno de los .jar.

✗ A mano · sin gestor de dependencias
// 1. Buscar cada .jar en internet // 2. ¿Qué versiones? ¿son compatibles? // 3. ...y las que CADA jar necesita // (las transitivas), también a mano // 4. Copiarlos a una carpeta /lib // 5. Decirle a Java dónde está cada uno // (eso es el classpath, escrito a mano): javac -cp ".;lib/sync.jar;lib/core.jar;lib/bson.jar" ...
Frágil y repetitivo. Si cambias de equipo, repites todo. Una versión mal elegida y nada compila. No escala.
✓ Con un gestor · declaras y él resuelve
<!-- En el pom.xml declaras solo el QUÉ --> <dependency> <groupId>org.mongodb</groupId> <artifactId>mongodb-driver-sync</artifactId> <version>5.8.0</version> </dependency> // Él trae esa Y sus transitivas.
Declaras el qué, no el cómo. El gestor resuelve versiones, trae las dependencias de tus dependencias y las cachea. Reproducible en cualquier equipo.

Eso es un gestor de dependencias: un programa que lee una lista de lo que tu proyecto necesita, lo descarga de un repositorio público (con todo lo que arrastra consigo) y lo deja listo para usar. Todo lenguaje moderno tiene el suyo — porque nadie reescribe lo que ya existe, y administrar ese préstamo a mano es imposible a escala.

Cocinar un plato no significa criar las gallinas ni moler el trigo: usas ingredientes que otros produjeron. Un gestor de dependencias es el servicio de entrega al que le pasas tu lista de la compra; él va al supermercado, trae cada ingrediente y los ingredientes que ese ingrediente necesita, y los guarda en tu despensa para la próxima vez.

II · Maven · el bibliotecario del mundo Java

Maven no solo trae librerías.
También sabe construir tu proyecto.

Apache Maven es el gestor de dependencias y herramienta de construcción más usado en Java. Nació hacia 2004 en la Apache Software Foundation: su creador, Jason van Zyl, estaba harto de mantener guiones de compilación casi idénticos y de descargar JARs a mano. Su idea fue "convención sobre configuración": si sigues una estructura estándar, casi no configuras nada. El nombre Maven viene del yidish y significa "acumulador de conocimiento" — la herramienta que sabe cómo construir tu proyecto y de dónde traer cada pieza.

La misma idea de la sección anterior, ahora con libros: Maven es un bibliotecario con un catálogo mundial. Le entregas una ficha (tu pom.xml) que dice qué necesitas; él lo busca en la biblioteca central (Maven Central, el supermercado de antes) y guarda una copia en tu despensa (~/.m2) para no volver a ir por ella.

Dos lugares que debes conocer. Maven Central es el repositorio público por defecto (lo navegas en central.sonatype.com); ahí viven millones de librerías, listas para que cualquiera las declare. Y ~/.m2/repository es tu repositorio local: la caché en tu propio equipo donde Maven guarda lo que ya bajó, para no repetir la descarga. Mira cómo resuelve una sola dependencia:

Y además construye: con un solo comando, mvn package hace lo que tú harías a mano en varios pasos — revisa el proyecto, lo compila, corre tus pruebas y arma el .jar final. Esos pasos son un ciclo de vida de fases en orden (validate → compile → test → package), y pedir la última dispara sola todas las anteriores.

V · El salto fuera de la memoria

Tus objetos vivían en RAM.
Al apagar, desaparecían.

Ya sabes traer cualquier librería sin descargarla a mano. La primera que usarás así —el driver de la base de datos— resuelve un problema aún más viejo que el de las dependencias. Porque todo lo que construiste en el Bloque I — clases, atributos, bucles, decisiones — vivía dentro del programa. Mientras corría, mandabas tú: una lista tenía lo que pusiste, un if elegía lo que decías. Pero al cerrar el programa, todo se borraba. Hoy das el salto que separa un ejercicio de un proyecto real: persistir datos — guardarlos en una base de datos para que sobrevivan al apagón.

Y aquí cambia una regla profunda. Una variable en memoria nunca falla. Una base de datos sí: el archivo puede estar bloqueado, la red caerse, la contraseña ser incorrecta, el dato venir con el tipo equivocado. Por primera vez tu código depende de algo que no controlas. Para eso existe el manejo de excepciones — el tema central de hoy.

Programar en memoria es cocinar en tu propia cocina: todo está donde lo dejaste. Hablar con una base de datos es pedir a domicilio: depende del restaurante, del repartidor y del tráfico. A veces llega tarde, a veces no llega. No puedes evitar que falle — pero sí puedes tener un plan B para cuando falle.

VI · La red de seguridad · try / catch / finally

El try es el salto.
El catch es la red que te atrapa.

Cuando una línea dentro de un try falla, no se rompe el programa: se lanza una excepción y el control salta al catch, que la atrapa. Esto rompe el flujo lineal que conoces — no se ejecuta línea por línea hasta el final, sino que salta. Avanza paso a paso con los dos escenarios y míralo:

Un trapecista salta sin miedo porque debajo hay una red. El try es el salto arriesgado; el catch es la red que lo recoge si cae. Sin red (sin try/catch), una sola caída termina la función. El finally es el utilero que recoge el equipo siempre — caiga o no caiga el trapecista.

Tres ideas que cambian todo: (1) la línea que falla corta el resto del try — lo de abajo no se ejecuta. (2) El catch solo corre si hubo error. (3) El finally corre siempre — por eso es el lugar para cerrar conexiones.

VII · La llave que se cierra sola · try-with-resources

Abrir una conexión es fácil.
Lo difícil es acordarse de cerrarla.

Una conexión a base de datos es un recurso caro y limitado: si la abres y olvidas cerrarla, se queda "colgada" y, tras muchas, la base se satura. Java tiene una forma de cerrar sola: declarar el recurso entre paréntesis del try (...). Al salir del bloque — con éxito o con error — Java llama a close() por ti.

✗ A mano · frágil
Connection c = null; try { c = DriverManager .getConnection(url); // ... usar c ... } catch (SQLException e) { // manejar error } finally { // hay que acordarse... if (c != null) c.close(); }
Funciona, pero tú cargas con cerrar: el finally, el if (c != null), y close() también puede fallar. Mucho ritual para no olvidarse.
✓ try-with-resources · seguro
try (Connection c = DriverManager .getConnection(url)) { // ... usar c ... } catch (SQLException e) { // manejar error } // c.close() se llama SOLO, // pase lo que pase
El recurso va dentro del paréntesis. Java garantiza close() al salir. Sin finally, sin if, sin olvidos. Es la forma que usarás siempre.
Es como un grifo con sensor: abre cuando pones las manos y se cierra solo al quitarlas. No dependes de tu memoria para no inundar el baño. El recurso (agua, conexión) se libera siempre, aunque salgas corriendo.

X · El cursor del ResultSet · tu bucle while reaparece

Un SELECT no te da los datos de golpe.
Te da un dedo que recorre filas.

El ResultSet no es una lista cargada en memoria: es un cursor que empieza antes de la primera fila. Cada rs.next() avanza una fila y devuelve true; cuando ya no hay más, devuelve false. Por eso se lee con un while — el mismo bucle que ya dominas. Pulsa rs.next() y observa:

idnombreedad
while (rs.next()) { int id = rs.getInt("id"); String n = rs.getString("nombre"); print(id + " " + n); }
El cursor está antes de la primera fila. Aún no has leído nada. Pulsa rs.next().

Fíjate en el cierre: cuando rs.next() devuelve false, el while termina solo. Es exactamente el patrón de bucle controlado por condición que viste con menús y validaciones — ahora aplicado a una base de datos real.

XI · El salto a la nube · MongoDB Atlas

La misma idea, otro mundo:
la base ya no es un archivo. Es un servidor.

SQLite era un archivo en tu disco. MongoDB Atlas es una base en internet: viven los datos en un servidor remoto al que llegas por red, con usuario y contraseña. Y cambia el modelo: en vez de tablas con filas, hay colecciones con documentos — fichas flexibles tipo JSON. El paquete cambia, pero el espíritu (abrir, operar, atrapar errores, cerrar) es el mismo. SQLite es un cuaderno en tu cajón: siempre está, es tuyo, nadie más lo toca. Atlas es un cuaderno en una bóveda al otro lado de la ciudad: necesitas la llave (usuario y contraseña), que el camino esté abierto (tu IP autorizada) y que haya señal — pero pueden consultarlo desde muchos lugares a la vez.

Un detalle clave: crear el MongoClient no conecta todavía — es como guardar un contacto en el teléfono. La conexión real (y el posible error de red) ocurre en la primera operación. Por eso se hace un ping al inicio: para fallar pronto y con un mensaje claro.

XII · Dos clases de excepción · ¿el compilador te obliga?

Algunas excepciones te las exige el compilador.
Otras, confían en ti.

Aquí está la diferencia más sutil entre los dos mundos. SQLException es checked: el compilador no te deja compilar si no la manejas o la declaras. MongoException es unchecked (hereda de RuntimeException): el compilador no dice nada… pero si la ignoras, un fallo de red tumba el programa igual.

SQLException · CHECKED · te obliga
// Esto NO compila: conn.prepareStatement(sql); // ⛔ "unreported exception // SQLException; must be caught" // El compilador EXIGE manejarla: try { conn.prepareStatement(sql); } catch (SQLException e) { ... }
Es un trámite con firma obligatoria: no avanzas sin atenderlo. Java te fuerza a pensar en el error antes de ejecutar.
MongoException · UNCHECKED · confía
// Esto SÍ compila: col.insertOne(doc); // ✓ el compilador no se queja // ...pero si la red falla y no // hay catch, el programa se cae. try { col.insertOne(doc); } catch (MongoException e) { ... }
Es una advertencia verbal: nadie te firma nada, pero ignorarla duele igual. La responsabilidad es tuya, no del compilador.
Una excepción checked es el cinturón de seguridad que pita hasta que te lo pones. Una unchecked es saber que conducir cansado es peligroso: nadie te obliga a descansar, pero el riesgo es real. Buen programador: maneja las dos, te obligue o no el compilador.
