El contrato.

Una interfaz no escribe código: escribe una promesa. Dice qué debe poder hacerse, jamás cómo. Hoy aprendes a prometer sin implementar — y a hacer que el lenguaje te obligue a cumplir.

I · Prometer sin implementar

Una interfaz es un contrato.
Define el qué, nunca el cómo.

Hasta hoy, cada clase decía qué hacía y cómo lo hacía. Una interfaz separa esas dos cosas: declara los métodos que algo debe tener — su firma, su nombre, lo que recibe y lo que devuelve — y deja el cuerpo vacío. Es una lista de obligaciones. Quien firme el contrato (quien lo implemente) está forzado a cumplir cada cláusula, o el programa ni siquiera arranca.

Piensa en un enchufe. La pared no sabe ni le importa si conectas una lámpara, un cargador o una licuadora. Solo exige una cosa: dos clavijas, este voltaje. Ese es el contrato. La lámpara y la licuadora lo cumplen de formas internas completamente distintas — pero ambas encajan. En software, esa pared es la interfaz: define la forma del enchufe, no lo que hay del otro lado.

En Java esto tiene una palabra dedicada: interface. En Python — nuestro lenguaje base — lo logramos con ABC (Abstract Base Class) y con Protocol. Tres caminos, la misma idea. Vamos a recorrerlos.

II · Anatomía de un contrato

Cada firma sin cuerpo
es una obligación que heredas.

Aquí está una ABC de Python: una pasarela de pagos. Fíjate que ningún método tiene cuerpo — solo ... o pass. Eso no es código incompleto: es una promesa exigida. Toca cada firma marcada para ver a qué te obliga si decides implementar este contrato:

from abc import ABC, abstractmethod class MetodoPago(ABC): # el contrato: QUÉ debe poder hacer # todo medio de pago. IV · Implementa o explota

Un contrato que no obliga
no es un contrato.

Aquí está la fuerza de una ABC: si firmas el contrato pero olvidas una cláusula, Python no te deja crear el objeto. El error salta al instanciar, no a la hora de usar el método. Recórrelo paso a paso — verás la clase incompleta, el intento de instanciarla y la explosión:

V · El castillo del encapsulamiento

No todo lo de un objeto
es para el mundo exterior.

Un buen contrato expone solo la puerta principal y esconde la maquinaria. Python tiene cuatro niveles de acceso, de la muralla más baja a la más alta. Ojo: en Python son convenciones y trucos, no candados de verdad como en Java — pero un profesional los respeta. Toca cada anillo del castillo:

saldo
Público
Sin guion bajo · cualquiera lo toca
_saldo
Protegido
Un guion bajo · "uso interno, por favor"
__saldo
Privado
Dos guiones · name mangling
@property
Propiedad
La puerta controlada · acceso con reglas
Nivel 0 · público

La estrella es @property: te deja escribir cuenta.saldo como si fuera un atributo simple, pero por debajo corre un método que valida, calcula o protege. Es el getter de Java (Sesión 3) con la cara amable de un atributo. Y en Java, el equivalente del __privado es la palabra real private — ahí sí el compilador te corta la mano.

VI · Adelanto — duck typing

Si camina como pato
y grazna como pato…
es un pato.

Aquí se cierra el círculo y se abre el siguiente. Una función que recibe "algo que sabe cobrar" no pregunta de qué clase eres — solo te pide que cumplas el contrato. Eso es duck typing, y es la antesala directa del polimorfismo del Sprint 2. Elige qué objeto le pasas a la misma función y mira qué pasa:

def procesar(metodo, monto): # no pregunto QUÉ eres, # solo pido que sepas cobrar() if metodo.cobrar(monto): return "Pago aprobado ✓" return "Pago rechazado ✗"

Taco()

VII · El taller de hoy — tu proyecto integrador

Diseña el contrato
de tu proyecto.

La experiencia de aprendizaje de hoy es práctica: vas a diseñar e implementar una interfaz o ABC para tu proyecto integrador, y a comparar cómo se vería en Java (interface) versus Python (ABC / Protocol). Estas preguntas te guían — las respuestas las decides tú según tu dominio:

Qué familia de objetos de tu proyecto comparte un mismo "deber hacer"?
Pista: busca varias clases que respondan a la misma pregunta de formas distintas.
Cuáles son las 2 o 3 cláusulas mínimas del contrato? (sus firmas)
Nombre del método, qué recibe, qué devuelve. Nada de cuerpo todavía.
Te conviene ABC (jerarquía + código común) o Protocol (encajar sin heredar)?
Si dos clases ya existentes deben "encajar" sin tocarlas, piensa en Protocol.
Qué atributos esconderías con _ o __, y cuál expondrías con @property?
El saldo, una clave, un identificador interno… ¿qué no debe tocar el exterior?
Reglas de la IA para diseñar tu contrato
✓Pídele que te explique: "¿cuándo conviene un Protocol en vez de una ABC?"
✓Pídele que critique tu diseño: "este es mi contrato Notificable, ¿le sobra o falta algún método?"
✓Pídele que te ayude a traducir tu ABC de Python a una interface de Java, para la comparativa
✗Pedirle "diseña la arquitectura de mi proyecto" — en la presentación defenderás cada cláusula: si no sabes por qué está ahí, no es tu contrato.

VIII · Para la próxima clase

Define el qué.
El cómo llega en el Sprint 2.

Hoy prometiste sin implementar. La próxima sesión vas a cumplir esas promesas de muchas formas distintas — eso es polimorfismo, y el contrato que diseñes hoy es exactamente la base que necesita.

Avance del proyecto · entregable de la sesión
  1. Diseña el contrato de tu proyecto integrador: una ABC o Protocol en Python con 2–3 métodos abstractos (solo firmas, sin cuerpo).
  2. Implementa al menos dos clases que firmen ese contrato, cada una cumpliendo las cláusulas a su manera.
  3. Aplica encapsulamiento avanzado: esconde lo interno con _ / __ y expón lo necesario con @property.
  4. Haz la comparativa: escribe cómo se vería ese mismo contrato como interface en Java. Una columna Python, una Java.
  5. Prepara tu presentación en clase: 3 minutos defendiendo tu contrato — qué obliga, por qué esas cláusulas y no otras.
  6. Deja lista la semilla del polimorfismo del Sprint 2: una función que reciba "cualquier cosa que cumpla el contrato".
  7. Sube todo a tu repo. El contrato bien diseñado hoy te ahorrará reescrituras la próxima semana.
