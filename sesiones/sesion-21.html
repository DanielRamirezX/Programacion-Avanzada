Las excepciones que ya conoces venían de fábrica. Hoy aprendes a nombrar lo que falla en TU proyecto: crear excepciones propias que digan, con sus palabras, qué salió mal y por qué.

I · Por qué fabricar tus propios errores

Un ValueError dice "algo está mal".
Un DatosInvalidosException dice qué.

Ya sabes atrapar excepciones con try / except y conoces las de fábrica: ValueError, KeyError, FileNotFoundError. El problema es que son genéricas: sirven para todo y no hablan de tu dominio. Cuando tu proyecto crece, necesitas errores que se llamen como tus reglas de negocio — para poder atraparlos por nombre y reaccionar distinto a cada uno. Eso es una excepción propia: una clase que hereda de Exception y le pone apellido a la falla.

II · La jerarquía — un árbol que crece

No una excepción suelta.
Una familia con un apellido común.

La clave del diseño es la raíz: una excepción base de tu proyecto (ProyectoException) de la que heredan todas las demás. Así puedes atrapar una sola y cazar a toda la familia, o atrapar una hija específica cuando te interesa el detalle. Pulsa Hacer crecer y toca cada nodo:

El apellido de la familia. Todas tus excepciones heredan de ella, así un except ProyectoException atrapa a cualquiera de tu proyecto sin tener que listarlas una por una.

III · Los mecanismos — Python ↔ Java

La misma idea
en dos idiomas.

Python es nuestro lenguaje base, pero la excepción propia existe en todos los lenguajes orientados a objetos — porque al final una excepción es solo una clase que hereda. Compara los cuatro movimientos clave. En Python heredas de Exception y lanzas con raise; en Java haces extends Exception y lanzas con throw:

V · Buenas prácticas — el mensaje importa

Una excepción no es solo un freno.
Es un mensaje a tu yo del futuro.

El nombre de la clase dice qué falló; el mensaje y los atributos dicen con qué datos falló. A las 3 de la mañana, depurando producción, la diferencia entre estos dos enfoques son horas de tu vida:

✗ Así NO
raise Exception("error") # genérica, sin contexto, # sin nombre de dominio
en consolaException: error
✓ Así SÍ
class DatosInvalidosException(Exception): def __init__(self, campo, valor): self.campo = campo self.valor = valor super().__init__( f"Campo '{campo}' inválido: {valor!r}")
en consolaDatosInvalidosException: Campo 'edad' inválido: -7
El nombre + el contexto = un error que se diagnostica solo. La versión rica guarda campo y valor como atributos: quien la atrapa no solo sabe que falló, sabe exactamente con qué dato falló — y puede mostrarlo, registrarlo o corregirlo.

VII · Puente a la práctica — "Cine éxito"

Una sala llena
también es una regla de negocio.

El taller de hoy conecta directo con la Práctica institucional 5. El caso "Cine éxito" gestiona funciones de cine, y hay una regla que el sistema debe defender por sí mismo:

Practicario MCT158 · Práctica 5 de 6 · requerimiento de excepción

El sistema de "Cine éxito" permite a los clientes comprar boletos para distintas funciones. Cada función tiene un número limitado de asientos disponibles. Cuando un cliente intenta comprar boletos para una función que ya no tiene asientos, el sistema no debe permitir la venta ni caer con un error genérico.

Se te pide crear la excepción personalizada AsientosAgotadosException para gestionar de forma controlada las funciones sin disponibilidad: la venta se rechaza con un mensaje claro, con el contexto de qué función y cuántos asientos se pidieron, y el flujo del programa continúa.

En nuestro proyecto integrador (Python), el taller es el mismo movimiento: armar la jerarquía ProyectoException → DatosInvalidos / RegistroNoEncontrado / AsientosAgotados e integrarla al código que ya tienes. Marca lo que tu jerarquía debe cumplir antes de presentarla en clase:

Buenas prácticas cubiertas: 0 de 8
Existe una raíz del proyecto (class ProyectoException(Exception)) de la que heredan todas.
Nombres descriptivos que terminan en Exception y dicen qué falló (no Error1, no MiExcepcion).
Cada excepción tiene un mensaje claro pasado a super().__init__(...).
Las que lo necesitan guardan contexto como atributos (campo, valor, id, función…).
Se lanzan con raise en el punto exacto donde se rompe la regla de negocio.
Se atrapan por su tipo específico, no con un except Exception que lo traga todo.
Quedan registradas con logging (nivel + contexto), no con print.
Integradas al proyecto existente: el código viejo ya las usa, no viven en un archivo aparte sin conectar.
