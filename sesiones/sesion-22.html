La memoria
que perdura.

Tus objetos viven en la RAM: brillan un instante y mueren cuando cierras el programa. Hoy le das a tu aplicación una memoria que sobrevive al apagón — la conectas a una base de datos y lees el primer dato desde código.

I · Por qué importa una base de datos

Sin persistencia, tu programa
tiene amnesia al cerrarse.

Todo lo que construiste hasta ahora — tus clases, tus objetos, tus listas — vive en la memoria volátil (RAM). Es rápida, pero se borra en cuanto el proceso termina. Una base de datos es la memoria que perdura: guarda en disco y, además, resuelve tres problemas que un archivo suelto no puede. Pulsa el interruptor:

Solo en memoria (RAM)
Programa encendido
Veloz, pero efímera: cierras el programa y todo se evapora.
En base de datos (disco)
Programa encendido
Más lenta de tocar, pero perdura: vuelves a encender y los datos siguen ahí.

Una base de datos profesional no solo guarda — aporta tres garantías que tu proyecto integrador va a necesitar:

Garantía 01
Persistencia
Los datos sobreviven al cierre del programa, al reinicio del equipo y al paso del tiempo. Viven en disco, no en RAM.
Garantía 02
Integridad
Reglas que impiden datos basura: tipos, claves, campos obligatorios. La BD rechaza lo que rompería tu modelo.
Garantía 03
Concurrencia
Muchos usuarios leyendo y escribiendo a la vez sin pisarse. La BD coordina los accesos por ti.

II · La cadena de conexión — su anatomía

Antes de hablar con la BD,
hay que saber su dirección.

El string de conexión (o connection string) es la dirección postal de tu base de datos: le dice al driver con qué motor hablar, en qué máquina vive, en qué puerto escucha y con qué credenciales entrar. Aquí una cadena de PostgreSQL — toca cada parte para ver qué significa:

postgresql://carlos:••••••@db.unitec.mx:5432/estancia_lujo
Driver
El motor
Indica qué sistema gestor (SGBD) usas: postgresql, mysql, sqlite… Determina qué driver toma la conexión.
¿Y SQLite? Su cadena es la más simple de todas: sqlite:///hotel.db o incluso solo el nombre del archivo "hotel.db". No hay host, ni puerto, ni usuario — porque no hay servidor: la base de datos es un archivo en tu disco. Por eso es perfecta para empezar.

V · El patrón Repository — objeto ↔ fila

Cada objeto de tu modelo
es una fila en una tabla.

Aquí está el puente entre tu POO y la base de datos: una instancia de tu clase corresponde a una fila; cada atributo, a una columna. Toca un atributo del objeto y mira a qué columna se mapea:

La ganancia: si mañana migras de SQLite a PostgreSQL, solo cambias el repositorio. Tus clases Habitacion y Reservacion ni se enteran. Eso es separar la lógica de datos de la lógica de negocio — la misma idea de encapsulamiento que ya dominas, ahora aplicada a la capa de persistencia.

El hotel "Estancia de lujo" quiere modernizar su sistema de reservaciones. Hoy todo se anota en libretas: se pierden datos, se duplican reservas y nadie sabe qué habitaciones están libres. Te contratan para construir el sistema con una base de datos real detrás.

El sistema manejará dos tablas: habitaciones (número, tipo, precio, disponibilidad) y reservaciones (huésped, habitación, fechas de entrada y salida).
