Todo o nada.

Una transferencia que resta de una cuenta pero no suma en la otra es peor que ninguna transferencia. Hoy cierras el CRUD con UPDATE y DELETE, y aprendes la regla que protege tus datos: commit si todo sale bien, rollback si algo falla.

I · La promesa

Una transacción es un trato:
se ejecuta completa, o no se ejecuta nada.

Hasta ahora hiciste INSERT (Create) y SELECT (Read). Hoy cierras el CRUD con las dos operaciones que modifican lo que ya existe: UPDATE y DELETE. Y justo porque modifican, necesitan una red de seguridad. Esa red se llama transacción: agrupa varias operaciones y exige que todas tengan éxito antes de quedar grabadas. Si una falla, se deshace todo con un rollback — como si nunca hubiera pasado. Es la propiedad de atomicidad: indivisible, todo o nada.

II · Todo o nada — la transacción en vivo

Transferir $500 son dos UPDATE:
restar de una cuenta, sumar en la otra.

Si el primero funciona y el segundo falla, el dinero se evapora. La transacción evita eso: las dos operaciones viven o mueren juntas. Pulsa Confirmar (commit) y se graban; provoca un error y mira cómo el rollback devuelve todo a su estado original. Ningún saldo a medias.

IV · Sin rollback — datos a medias

El bug más caro no es el que truena.
Es el que deja la base mintiendo.

Un programa que se cae se nota. Pero una transferencia que restó y no sumó deja datos silenciosamente corruptos: el programa siguió, nadie vio el error, y la base ahora afirma algo falso. Mira la misma falla en las dos columnas — sin red de seguridad a la izquierda, con rollback a la derecha:

Ana$1500
Luis$300
Total en el banco: $1800 ✓
Cada UPDATE se graba al instante. Nadie vigila el conjunto.
✓ Con transacción + rollback
Ana$1500
Luis$300
Total en el banco: $1800 ✓
Nada se graba hasta el commit. Si algo falla, rollback.

V · El patrón blindado — Python ↔ Java

try → ejecuta → commit.
except → rollback → relanza.

Aquí está el molde que vas a copiar en cada operación de escritura de tu proyecto. La idea es idéntica en los dos lenguajes; cambia la sintaxis. Recorre las pestañas: el patrón en Python (tu lenguaje base), el equivalente en Java, y un DELETE seguro con su WHERE:

Y el detalle que separa un UPDATE/DELETE profesional de un desastre: el WHERE. Olvidarlo no da error — hace algo mucho peor:

✗ Lo que jamás debes escribir
cur.execute( "DELETE FROM reservas" ) cur.execute( "UPDATE cuentas SET saldo = 0" )
Sin WHERE, afecta TODA la tabla. Un DELETE sin condición borra cada reserva del hotel. SQL te obedece sin chistar — no hay "deshacer" después del commit.
✓ Lo que sí, siempre
cur.execute( "DELETE FROM reservas WHERE id = ?", (reserva_id,) # parámetro ) if cur.rowcount == 0: raise ValueError("no existe")
WHERE acota · ? parametriza · rowcount verifica. Revisa rowcount: si es 0, no afectaste nada — probablemente el id no existía, y eso merece una excepción.
