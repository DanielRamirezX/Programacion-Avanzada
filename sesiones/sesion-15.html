II · El caso — lee antes de programar

Un sistema que no solo califica:
retroalimenta.

Practicario MCT158 · Práctica 2 de 6 · Sentencias de control y repetición

Formas parte de un equipo de desarrollo de software para instituciones de educación. A tu equipo se le asignó implementar un sistema para clasificar y dar retroalimentación sobre el desempeño académico de los estudiantes. El sistema se construirá con orientación a objetos en Java, usando las sentencias de control if, if/else e ifs anidados.

Con base en la nota final de cada estudiante, el sistema deberá: evaluar si aprobó o no; clasificar su desempeño en niveles — excelente, bueno, regular o reprobado; y mostrar un mensaje motivacional o de mejora según el nivel obtenido.

Requerimiento 01
Evaluar el estado
Con la nota final, decidir si el estudiante aprobó o reprobó. Una sola decisión → un if/else.
Requerimiento 02
Clasificar el desempeño
Ubicar la nota en un nivel: excelente, bueno, regular o reprobado. Varias decisiones encadenadas → ifs anidados.
Requerimiento 03
Mensaje motivacional
Según el nivel, mostrar un mensaje de felicitación o de mejora. La retroalimentación es el corazón del sistema.

El practicario no fija la escala: tu equipo la define y la defiende. En esta guía usaremos la convencional — aprobado con >= 60; niveles: 90–100 excelente, 80–89 bueno, 60–79 regular, < 60 reprobado. Si tu equipo elige otra, documenta el porqué en la historia de usuario.

III · El concepto — tres formas de decidir

Un programa sin decisiones
siempre cuenta la misma historia.

Toda la práctica gira sobre tres estructuras. El if decide si algo pasa o no. El if/else elige entre dos caminos. Y el if anidado mete una decisión dentro de otra — primero filtro grueso, luego filtro fino. Elige una pestaña y pulsa ▶ Recorrer para ver, línea por línea, qué camino toma Java con una nota de ejemplo:

Pulsa ▶ Recorrer para animar la ejecución paso a paso.

Regla de lectura: en Java la condición siempre va entre paréntesis y el bloque entre llaves. No hay elif como en Python — se escribe else if, con espacio. Y la indentación en Java no cambia el significado (las llaves mandan), pero sí cambia la legibilidad: indenta como si fuera Python.

Ejemplo de referencia · HU-01
Como docente de la institución, quiero registrar la nota final de cada estudiante y que el sistema determine su estado, clasifique su desempeño y muestre un mensaje personalizado, para dar retroalimentación inmediata que motive la mejora académica.
Criterios de aceptación
  • La nota final es un número entre 0 y 100.
  • El estado es Aprobado si la nota es >= 60; en otro caso, Reprobado.

6
Implementar evaluarEstado() — el if/else

Traduce el diagrama del paso 3 a código: el rombo se vuelve la condición, cada rama se vuelve un bloque. Un solo if/else, dentro de la clase Estudiante:

7
Implementar la clasificación — ifs anidados

Ahora el diagrama del paso 4: primero la decisión gruesa (¿aprobó?) y, dentro del sí, la cascada fina de niveles. Ese if-dentro-del-if es el if anidado del tema 3.2. Completa con el mensaje y un Main que lo pruebe:

VII · Cómo se evalúa — la rúbrica

Tres frentes.
El mismo estándar.

Frente 01
Análisis del caso
  • Identificación adecuada de actores, objetivos y restricciones.
  • Historia de usuario clara y completa: rol, acción y propósito.
Frente 02
Diseño del sistema
  • Diagramas de actividades con UML correcto: rombos para decisiones, rectángulos redondeados para actividades, flechas de transición.
  • Los diagramas reflejan las decisiones del programa (if, if/else, anidados).
  • Coherencia entre diagramas e historia de usuario.
Frente 03
Implementación en Java
  • Clases con los atributos y métodos necesarios, bien codificados.
  • POO real: modificadores de acceso y código dividido en unidades lógicas.
  • Programa funcional que cumple los tres requerimientos.

La regla de oro de la Sesión 8 aplica entera: el diagrama y el código deben contar la misma historia. Si tu rombo dice >= 60 y tu if dice > 60, ya perdiste puntos — y un estudiante con 60 exacto, su aprobación.
