El cruce
de caminos.

Hasta hoy tu código corría en línea recta. A partir de hoy decide: toma un camino u otro según lo que encuentre. Esa bifurcación — el if — es la primera vez que tu programa piensa.

I · La primera decisión

Un programa sin condiciones
solo sabe obedecer. Con ellas, empieza a elegir.

Una sentencia de control evalúa una pregunta de sí o no — una condición booleana — y según la respuesta, ejecuta un bloque u otro. Es el cruce de caminos del temario 3.1: if abre el primer camino, elif añade desvíos intermedios, y else es el camino que queda cuando ninguno se cumple. Hoy lo aprendemos en Python — el lenguaje base del curso — y miramos de reojo cómo lo dicen Java, JavaScript y Go.

II · El flujo, paso a paso

El if no prueba todo:
se detiene en el primer camino que se cumple.

Esta es la regla que más se malentiende: Python evalúa las ramas en orden, de arriba abajo, y en cuanto una es verdadera ejecuta su bloque y abandona la cadena completa — ni siquiera mira las de abajo. Suelta una calificación y observa qué rama toma y cuáles ni se prueban:

—
if nota >= 90: return "Excelente" elif nota >= 70: return "Bueno" elif nota >= 60: return "Regular" else: return "Reprobado"

Pulsa una calificación abajo. La nota caerá y se irá probando contra cada rama, en orden.

¿Notaste el orden? Si pusieras nota >= 60 arriba del todo, todas las notas aprobatorias caerían ahí y nunca llegarías a "Excelente". En una cadena if/elif el orden no es estético: es lógica. De lo más restrictivo a lo más general.

IV · De la pirámide a la línea recta

El if anidado es fácil de escribir
y un infierno de leer.

Cuando metes condiciones dentro de condiciones, el código se va recorriendo a la derecha — la temida "pirámide de la perdición". Las guard clauses (cláusulas de guarda) le dan la vuelta: validas lo que no debe pasar al inicio, sales temprano con return, y el camino feliz queda al final, sin sangría. ✗ Lo que produce pirámides
  • if dentro de if dentro de if para validar precondiciones.
  • Un else gigante al final que nadie alcanza a leer.
  • Mezclar validación (¿es válido?) con la lógica real en el mismo bloque.
✓ Lo que aplana el código
  • Early return: sal en cuanto sepas la respuesta. No esperes al final.
  • Guard clauses: "si falta el dato, regreso ya" — al inicio.
  • El camino feliz al final, alineado a la izquierda, sin sangría.
