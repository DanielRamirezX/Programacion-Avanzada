Dos padres,
un orden.

Cuando una clase hereda de dos a la vez, surge la pregunta: ¿a quién le hace caso primero? Python no improvisa — tiene un orden exacto, el MRO. Hoy aprendes a leerlo y a respetarlo.

I · El problema de tener dos padres

Heredar de uno es fácil.
Heredar de dos abre una pregunta nueva.

Hasta ahora una clase hija tenía un padre y todo era una línea recta: si la hija no tiene un método, sube al padre. Pero Python permite herencia múltiple: una clase puede heredar de varias a la vez. Y entonces, si dos abuelos definen el mismo método saludar()… ¿cuál gana? La respuesta no es "el primero que se te ocurra" — es un orden formal llamado MRO (Method Resolution Order). Hoy lo desarmamos, vemos cómo super() lo recorre, y por qué Java prefirió prohibir todo esto.

II · El MRO — la linealización C3

Python toma el diamante
y lo aplana en una fila ordenada.

El diamante clásico, contado en familia: el Hijo hereda del Padre y de la Madre; ambos heredan del Abuelo. Cuando llamas un método en un objeto Hijo, Python no se pierde: aplica el algoritmo C3 y produce una sola lista de búsqueda. Pulsa para ver cómo ilumina la clase en cada paso, y fíjate en la regla de oro: el Abuelo va al final, después de todos sus descendientes.

Pulsa Siguiente para que Python busque el método empezando por Hijo. Verás el orden exacto que sigue.
Lo que verías en la terminal
>>> Hijo.__mro__  te devuelve la tupla (Hijo, Padre, Madre, Abuelo, object). Toda clase termina en object — es la raíz universal de Python. El MRO no es magia: es una propiedad que puedes inspeccionar en cualquier momento con Hijo.__mro__ o Hijo.mro().

III · super() — no es "el padre", es "el siguiente"

El error más común:
creer que super() llama al padre.

Falso. super() llama a la siguiente clase en el MRO — que no siempre es un padre directo. En herencia múltiple, si cada clase hace su trabajo y luego llama a super(), el control viaja por toda la cadena, en orden, sin saltarse a nadie. A esto se le llama cooperación: cada clase coopera pasando el testigo. Pulsa Play y mira cómo el saludo recorre las cuatro clases del diamante.

Hijo.greet() → "soy el Hijo" + super()
Padre.greet() → "soy el Padre" + super()
Madre.greet() → "soy la Madre" + super()
Abuelo.greet() → "soy el Abuelo" (fin de la cadena)
consola — pulsa Play o Siguiente…
Cada clase imprime lo suyo y llama a super(), que consulta el MRO para saber a quién le toca. Nadie llama a "su padre": llaman al siguiente de la fila.
La regla práctica
Si todas las clases de la jerarquía llaman a super().__init__(...) en su constructor, cada clase se inicializa exactamente una vez y en el orden del MRO. Si una sola rompe la cadena (no llama a super), las que vienen después en el MRO nunca se ejecutan. Por eso la disciplina importa: en herencia múltiple, super() es un contrato colectivo.

IV · Python ↔ Java — dos filosofías

Python dice: "confía en el orden".
Java dice: "mejor no te dejo".

Java no permite heredar de dos clases a la vez (no hay extends A, B). En su lugar, una clase implementa varias interfaces: contratos sin estado compartido, donde el conflicto del diamante casi desaparece. V · El problema del diamante

El mismo abuelo, heredado
por dos caminos. ¿Quién manda?

Este es el clásico que durante décadas hizo que muchos lenguajes le tuvieran miedo a la herencia múltiple. El Padre y la Madre heredan ambos del mismo Abuelo, y el Hijo hereda de los dos — así que lo del Abuelo le llega por dos caminos. Si el Abuelo se ejecutara dos veces, o si nadie supiera cuál versión usar, sería un caos. (Sí: en este árbol papá y mamá comparten abuelo. VI · Frenos a la herencia — final y sealed

A veces lo correcto
es decir: "de aquí no pasas".

Heredar es poderoso, pero no todo debe heredarse. Hay clases y métodos que no deben extenderse ni sobrescribirse — por seguridad, por contrato o porque su comportamiento es definitivo. Java lo expresa con final y sealed; Python, fiel a su estilo, lo hace por convención y con ayudas. Pulsa cada concepto:

final class
Clase no heredable
final method
Método no sobrescribible
sealed
Herencia restringida
¿Y en Python?
Python no tiene final a nivel de lenguaje: confía en que el equipo respete las reglas ("somos todos adultos aquí"). Pero hay herramientas: el decorador @final del módulo typing avisa al verificador de tipos (mypy) si alguien intenta heredar o sobrescribir, y __init_subclass__ permite bloquear la herencia en tiempo de ejecución. La filosofía es la misma de siempre: Python te advierte, Java te detiene.

VII · Práctica guiada — herencia que se ve

Hasta ahora el MRO era un diagrama.
Ahora va a moverse en tu pantalla.

Vamos a construir, paso a paso, una mini-app con tkinter — la biblioteca gráfica que ya viene con Python (nada que instalar). ¿Por qué tkinter en la clase de herencia? Porque en tkinter todo widget es una clase y la forma natural de usarlo es heredando. Y porque nos permite conocer el mixin: el uso legítimo más común de la herencia múltiple en el mundo real. Cada paso es un archivo listo para correr en Laboratorio/sesion-16/ — escribe, corre, observa, y hasta entonces pasa al siguiente.

Mixins en el mundo real
Los mixins son la razón por la que la herencia múltiple sigue viva. La biblioteca estándar los usa: socketserver.ThreadingMixIn convierte cualquier servidor en multihilo con una línea — class MiServidor(ThreadingMixIn, TCPServer). Django los usa para permisos: class MiVista(LoginRequiredMixin, View). La receta es siempre la misma: el mixin primero, la clase "real" después, y el nombre termina en Mixin para que se lea la intención.

IX · Para la próxima clase

La herencia bien diseñada no se nota.
La mal diseñada, se sufre.

Tu proyecto integrador ya tiene clases sueltas. Es hora de darles una jerarquía: qué hereda de qué, dónde coopera super(), y qué clases conviene dejar como base para lo que viene — porque la próxima unidad son interfaces y clases abstractas (ABC), y se construyen sobre lo de hoy.

Commit de la jerarquía de herencia
  1. Diseña la jerarquía de tu proyecto: identifica qué clases comparten comportamiento y modélalo con herencia (múltiple solo donde de verdad aporte — el Administrador es buen candidato, no fuerces diamantes).
  2. Cadena de super() correcta: cada __init__ que herede debe llamar a super().__init__(...). Verifica que cada clase se inicializa una sola vez.
  3. Imprime y comenta el MRO: agrega una línea que muestre TuClase.__mro__ y un comentario que explique por qué ese orden.
  4. Prepara el terreno para ABC: marca qué clases serán base (no instanciables directamente) — la próxima sesión las volveremos abstractas con abc e interfaces.
  5. Reto opcional — un mixin en tu proyecto: identifica una habilidad transversal (auditable, exportable a JSON, notificable…) y modélala como mixin, como el SacudibleMixin de la práctica guiada. Recuerda: el mixin va primero en la lista de bases.
  6. Sube el commit a GitHub con la jerarquía y el MRO documentado. Es tu participación de la sesión — ese commit es la evidencia.
