Es la puerta del sistema. Recibe peticiones HTTP, valida la entrada, llama a tu lógica y devuelve JSON. No sabe SQL ni sabe de IA: solo orquesta.
conecta con → lógica OOP (le pide hacer cosas) · módulo IA (en /recomendar)

La capa de entrada existe en todos los lenguajes — solo cambia la herramienta. Nosotros usamos FastAPI porque Python es la base del curso y porque convierte una función en un endpoint con un solo decorador. Mira la misma idea, "una ruta que saluda", en tres stacks:

Python · FastAPI
@app.get("/") def raiz(): return {"ok": True}
Lo que usas hoy. El decorador es la ruta. Tipos de Python + Pydantic te dan validación y docs automáticas en /docs.
JavaScript · Express
app.get("/", (req, res) => { res.json({ ok: true }); });
El estándar de Node. Mismo concepto — método, ruta y handler — con callbacks (req, res) en lugar de un valor de retorno.
Go · net/http
http.HandleFunc("/", func(w, r) { fmt.Fprint(w, "ok") })
Compilado y veloz. Más verboso, pero la misma anatomía: registras una ruta y escribes la respuesta. Lo que aprendes hoy se traduce solo.

IV · Errores con sentido — excepción propia → código HTTP

Un error no es un crash.
Es un mensaje con número.

Tus clases ya lanzan excepciones propias — las creaste en la unidad de manejo de errores. El truco de una buena API es traducir cada una al código HTTP correcto: así el cliente sabe si la culpa fue suya (4xx) o tuya (5xx). Toca una excepción y mira a qué código se mapea:

RegistroNoEncontradoException
Pediste el producto 999 y no existe en la base.
DatosInvalidosException
Mandaste un precio negativo o un nombre vacío.
Exception (no controlada)
Se cayó la base de datos o un bug inesperado.
404
Not Found
El recurso no existe.

En FastAPI esto se hace en un solo lugar con un @app.exception_handler: capturas tu excepción de dominio y devuelves el JSONResponse con el status correcto. Tu lógica de negocio nunca habla de HTTP — solo lanza excepciones limpias.

VI · Prompt engineering — aplicado a tu API

El modelo es el mismo.
El resultado depende del prompt.

Dentro de tu endpoint, el prompt es código: lo construyes con f-strings y los datos del sistema. Un prompt pobre da respuestas inservibles para una API; uno bueno da salida predecible y parseable. Compara los dos enfoques en el mismo endpoint /recomendar:

Prompt pobre ✗
"recomiéndame algo"
Sin rol, sin contexto, sin formato. No le dices quién es, qué datos tiene ni cómo quieres la respuesta. La IA improvisa — y cada llamada contesta distinto.
→ respuesta en prosa libre, imposible de convertir a JSON sin romperse
Prompt diseñado ✓
Eres un asesor de ventas. El cliente compró: {historial}. Del catálogo {productos}, sugiere 1 producto y devuelve solo JSON: {"id": int, "razon": str}.
Rol + contexto + formato explícito. Le das un papel, le inyectas tus datos reales y le exiges la estructura exacta que tu API necesita.
→ {"id": 7, "razon": "complementa su última compra"} · listo para devolver
La regla de oro · si la IA lo hizo, tú lo defiendes
✓Pídele que te explique un código HTTP: "¿cuándo conviene 422 en vez de 400?"
✓Pídele que revise tu prompt: "¿este prompt garantiza salida JSON válida?"
✓Pídele que te ayude a leer un error de FastAPI que no entiendes
✗Pedirle "hazme toda la API con IA" — en el Demo Day vas a defender tu código línea por línea; si no lo entiendes, no es tuyo.
