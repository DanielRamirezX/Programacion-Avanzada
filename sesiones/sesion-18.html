Sesión XVIII · Bloque II · Polimorfismo · Sprint 2 Review · Ref. Temario 5.6 · 5.6.1 · 5.6.2 · 5.6.3 · Python · Java

Un mensaje,
muchas respuestas.

Le mandas el mismo mensaje a una lista de objetos distintos — y cada uno responde a su manera. Eso es polimorfismo: muchas formas de cumplir un mismo contrato. Hoy lo implementas, y presentas tu Sprint 2.

I · La palabra, partida en dos

Poli = muchas.
Morfo = formas.

Ya dominas herencia (sesiones 15-16) y interfaces (sesión 17). El polimorfismo es lo que les da sentido: cuando varias clases comparten un método con el mismo nombre pero cuerpo distinto, puedes tratarlas a todas igual y dejar que cada una responda según lo que es. Escribes el código una vez; funciona para clases que aún ni existen.

El contrato
Todas prometen un método: area(), sonido(), dibujar(). Viene de una clase base o de una interfaz.
La sobrescritura
Cada subclase reimplementa el método a su manera. En Python es directo; en Java lo marcas con @Override.
El despacho
En tiempo de ejecución, el objeto real decide qué versión corre. No el tipo de la variable: el objeto.

II · La colección heterogénea

Una lista con figuras distintas.
Un solo for. Cada una responde lo suyo.

Aquí está el corazón del tema. Tienes una lista con un círculo, un cuadrado, un triángulo y un rectángulo — tipos diferentes en la misma colección. Pulsa ejecutar y observa: el mismo figura.area() recorre la lista, pero cada objeto calcula su área a su manera. Eso es un argumento polimórfico en acción.

●
Circulo(r=3)
subclase de Figura
■
Cuadrado(l=4)
subclase de Figura
▲
Triangulo(6,2)
subclase de Figura
▬
Rectangulo(5,2)
subclase de Figura
El for no pregunta de qué tipo es cada figura. Solo le manda el mensaje area() y confía en que cada objeto sabe responder. Ese es el regalo del polimorfismo: código que no cambia cuando agregas figuras nuevas.

III · La sobrescritura — Python ↔ Java

Mismo nombre de método.
Cuerpo reescrito en cada hija.

Para que el for de arriba funcione, cada subclase sobrescribe (override) el método de la base. En Python solo redefines el método; en Java agregas la anotación @Override — que no es obligatoria, pero el compilador la usa para protegerte. Recorre los seis pasos en ambos lenguajes:

Python tu lenguaje base
Java polimorfismo formal

IV · Duck typing — el sello de Python

«Si camina como pato
y suena como pato…
es un pato.»

Java exige que el contrato sea formal: para meter algo en una lista de Volador, la clase debe declarar que implementa esa interfaz. Python no pregunta el linaje: si el objeto tiene el método que necesitas, lo usa y punto. No importa de qué clase venga. Toca cada objeto y comprueba si "vuela":

🐦
Pajaro
def volar(self): ...
✈️
Avion
def volar(self): ...
🦸
Superheroe
def volar(self): ...
🐧
Pinguino
def nadar(self): ...
🪨
Piedra
# sin métodos útiles
El código solo pregunta:
¿tienes .volar()?
El Pajaro, el Avion y el Superheroe no comparten ninguna clase base — y aun así los tres vuelan en el mismo for. En Java tendrías que obligarlos a implementar Volador; en Python basta con que hagan lo que se espera. Poder y peligro a la vez: flexibilidad sin red.

V · isinstance / instanceof — el bisturí

A veces preguntar el tipo es necesario.
Casi siempre es señal de mal diseño.

Python tiene isinstance(obj, Clase) y Java tiene obj instanceof Clase: ambos preguntan "¿este objeto es de este tipo?". Son útiles — pero si tu código se llena de if isinstance(...) para decidir qué hacer, casi siempre estás reinventando el polimorfismo a mano. Revisa cada caso y descubre cuándo está bien y cuándo huele mal:

if not isinstance(monto, (int, float)):
✓ caso legítimo · validar entrada
if isinstance(dato, dict): ... else: ...
✓ caso legítimo · frontera de datos
if isinstance(fig, Circulo): area = ...
✗ olor · despacho manual
if isinstance(a, Perro): ... elif isinstance(a, Gato): ...
✗ olor · cadena de tipos
Regla práctica de diseño
✓Usa isinstance en la frontera del programa: validar lo que entra, lo que llega de un archivo o de la red.
✓Úsalo cuando un objeto puede tener tipos verdaderamente distintos que no comparten contrato (ej. str vs list).
✗Si lo usas para elegir qué método llamar según la clase — borra el if y deja que cada objeto responda. Eso es el polimorfismo trabajando por ti.

VI · El ejercicio de hoy — el zoológico

Una lista de animales.
Un .sonido(). Cada quien, su voz.

Antes del Sprint Review, construyes juntos el caso clásico — pero en tu propio dominio. La idea es idéntica a las figuras: una colección heterogénea, un mismo mensaje, respuestas distintas. Tú eliges el dominio (animales, instrumentos, medios de transporte, notificaciones). Esta es la pauta:

1 · El contrato
Una base Animal (clase abstracta o Protocol) que promete sonido(). Repasa lo de la sesión 17.
2 · Las formas
Tres o cuatro subclases — Perro, Gato, Vaca — que sobrescriben sonido() cada una.
3 · El polimorfismo
Una lista [Perro(), Gato(), Vaca()] y un for a in animales: a.sonido(). Cero if de tipo.
Reto extra para quien quiera empujar: agrega un Pato que además nade, y muestra cómo el for de sonidos lo trata igual que a los demás sin tocar una sola línea existente. Eso es lo que demuestra que tu diseño es polimórfico de verdad: extensible sin modificar.
